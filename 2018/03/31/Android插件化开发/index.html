Android插件式开发

Android
  • Android
  • 插件式开发

TIP: 插件框架:DynamicLoadApk, DynamicApk

要解决的痛点

  1. 项目越来越大，下载安装速度变长
  2. 各业务集中在一起
  3. 代码耦合严重
  4. AS编译速度越来越慢
  5. 方法数超过65535

插件式开发带来的优势

  1. 客户端热部署能力
  2. 编译速度提升，不再需要去放个风回来等编译结束了
  3. 解耦合解耦合解耦合，重要的事情说三遍
  4. app首次启动速度提升
  5. 业务模块按需加载，减少安装时间
     注：multidex技术可以解决方法数超过65535个的问题上，dalvik虚拟机在首次启动app时会进行dex的初始化，将dex转为opt，但是在ART上由于安装时，就进行了dex2aot，所以首次启动的等待时间长的问题不存在了


下面介绍一下途牛使用的DynamicLoadApk这个插件式开发框架，这个插件化框架是任玉刚和团队写的

DynamicLoadApk

1.加载插件的准备
 首先插件apk并没有安装，而是改成so文件放在lib文件夹中，那么它其中的组件和资源文件是如何被系统识别并加载的呢？
 在加载插件之前首先做一些准备工作，我们都知道在android中加载普通类的类加载器主要有两种PathClassLoader和DexClassLoader，它们两个的区别就在于PathClassLoader只能加载系统已经安装了的apk的指定目录下的类，而DexClassLoader还可以加载未安装的apk，zip等其中的类，那么这其中我们就需要准备一个插件所在目录地址下的DexClassLoader实例了
第一步：new一个给了指定目标地址的DexClassLoader 
private DexClassLoader createDexClassLoader(String dexPath) {
    File dexOutputDir = new File(mContext.getFilesDir().getAbsolutePath(), "plugin_dex");
    if (!dexOutputDir.exists()) {
        dexOutputDir.mkdirs();
    }
    if (dexOutputDir.exists()) {
        return new DexClassLoader(dexPath, dexOutputDir.getAbsolutePath(), mNativeLibDir, mContext.getClassLoader());  
    } else {
        return new DexClassLoader(dexPath, mContext.getFilesDir().getAbsolutePath(), mNativeLibDir, mContext.getClassLoader());
    }
}
 另外插件中的资源文件如何加载呢？我们都知道我们想让系统加载资源文件都是通过AssetManager这个类去管理，这个类中有个隐藏的方法addAssetPath，从方法名中就能看出，是加载资源的目录，那么我们反射方式调用这个方法，给它一个插件目录地址，那么就可以加载插件目录中的资源文件了。再把这个AssetManager传递给Resource
第二步：创建加载指定目录资源文件的Resources 
private AssetManager createAssetManager(String dexPath) {
    AssetManager assetManager = AssetManager.class.newInstance();
    Method addAssetPath = assetManager.getClass().getMethod("addAssetPath", String.class);
    addAssetPath.invoke(assetManager, dexPath);
    return assetManager;
}

private Resources createResources(AssetManager assetManager) {
    Resources superRes = mContext.getResources();
    Resources resources = new Resources(assetManager, superRes.getDisplayMetrics(), superRes.getConfiguration());
    return resources;
}
 将创建好的DexClassLoader和Resources实例放入插件的DLPluginPackage中，这个插件package中封装了加载同一个插件apk类和资源DexClassLoader，AssetManager，Resouces，另外还有PackageInfo等一些成员变量，至此，准备工作完成了
第三步：创建插件Package 
pluginPackage = new DLPluginPackage(dexClassLoader, resources, packageInfo, pluginInterface, realPath);
2.加载插件
 那么如何启动插件的Activity并且管理它的呢，原理很简单，就是使用一个代理Activity去代理执行插件Activity(插件中的所有组件其实就是一个普通的java类，没有系统上下文管理它们)
第一步：启动插件Activity实际是启动一个代理Activity 
public int startPluginActivityForResult(Context context, DLIntent dlIntent, int requestCode) {
    String packageName = dlIntent.getPluginPackage();
    DLPluginPackage pluginPackage = mPackagesHolder.get(packageName);
    final String className = getPluginActivityFullPath(dlIntent, pluginPackage);
    Class<?> clazz = loadPluginClass(pluginPackage.classLoader, className);
    Class<? extends Activity> activityClass = getProxyActivityClass(clazz);
    dlIntent.setClass(context, activityClass);
    dlIntent.setData(Uri.parse(new StringBuilder()
            .append(DLConstants.PLUGIN_SCHEMA).append(":")
            .append(packageName).append(":").append(className).toString()));
    performStartActivityForResult(context, dlIntent, requestCode);
    return START_RESULT_SUCCESS;
}
 代理Activity中持有DLProxyImpl实例，在代理Activity的onCreate方法中调用DLProxyImpl的onPreCreate方法用反射方法创建插件Activity实例并绑定到代理Activity中，这样在系统管理代理Activity的时候，代理Activity就可以同时调用插件Activity的生命周期方法，达到系统管理的效果
第二步：在代理activity创建时反射创建插件activity并代理它的生命周期 
public int onPreCreate(Intent intent) {
    Uri uri_data = intent.getData();
    String uri_str = uri_data.toString();
    String[] info = uri_str.split(":");
    mPackageName = info[1];
    mClass = info[2];
    mPluginManager = DLPluginManager.getInstance(mProxyActivity);
    mPluginPackage = mPluginManager.getPackage(mPackageName);
    mAssetManager = mPluginPackage.assetManager;
    mResources = mPluginPackage.resources;
    intent.setExtrasClassLoader(mPluginPackage.classLoader);
    initializeActivityInfo();
    handleActivityInfo();
    //通过创建和销毁来模拟实现task的管理
    int result = mPluginManager.setLauncheMode(mProxyActivity, mClass, mActivityInfo.launchMode);
    Class<?> localClass = getClassLoader().loadClass(mClass);
    Constructor<?> localConstructor = localClass.getConstructor(new Class[]{});
    Object instance = localConstructor.newInstance(new Object[]{});
    mPluginActivity = (DLPlugin) instance;
    ((DLAttachable) mProxyActivity).attach(mPluginActivity);
    // attach the proxy activity and plugin package to the mPluginActivity
    mPluginActivity.attach(mProxyActivity, mPluginPackage.packageName);
    return 0;
}
 以上就大致解释了DynamicLoadAPK的框架原理，这个框架思路很巧妙，使用代理方式去管理插件apk中的组件，但是需要再框架中实现了的组件才可以用，比如android的四大组件，目前只实现了activity，service这两种，而广播和contentProvider由于使用频率不高并没有在框架中实现，而且使用的时候是需要遵守一些使用的约束的，所以各业务新来的小伙伴需要一定的培训去了解一下这个插件化开发框架的原理才能更好的进行业务的开发
插件的开发需要遵循一定的约束，比如：
Activity需要继承自DLBasePluginActivity
使用that代替this使用，因为插件Activity只是一个普通的java类，并没有上下文
插件在开发中需要使用到的公共类库中方法，在插件gradle中使用provided提供编译支持，但是不打包进入apk中



另外简单介绍一下携程的DynamicApk这个插件式开发框架

DynamicApk

1.资源的编译和加载
 众所周知，android在打包过程中，使用aapt进行资源的打包，由于历史原因携程的插件中需要使用主工程的资源文件，那么就需要插件资源去依赖主工程apk，而这一点是可以做到，在aapt打包的过程其实是依赖了android.jar包中的android资源的，比如你可以通过@android:color/opaque_red形式来引用SDK自带资源
第一步：修改aapt命令参数，引用主工程资源
 打包资源文件后会生成R.java文件，去定义Android程序中所有各类型的资源的索引,所有id由三个字节表示，第一字节代表了Package，第二字节为分类，三四字节为类内ID
第二步：修改头字节给不同apk分配不同packageId,避免冲突
 同样是使用反射方式修改AssetManager中的addAssetPath方法中的path路径，那么是如何将这个自定义的资源管理类去接管系统的资源管理类呢，那么就需要在系统创建组件的时机去hack系统的方法，这个框架中就hack了Instrumentation类和在创建activity时换取了自定义的Resources类
2.类的加载
 如果你了解google的MultiDex的加载方式，那么对这个框架类的加载就不陌生了，它也是修改了android的类加载器PathClassLoader其中的成员变量pathList，并且将插件的dex的目录添加到这个成员变量的Elements数组中，这样，查找类的时候就会遍历所有的数组去加载类
